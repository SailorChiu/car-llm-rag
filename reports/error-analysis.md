# 指令解析 SFT:评测协议与错误驱动迭代

> 任务:中文车控语音指令 → 严格 JSON 函数调用(固定 10 个车控 API)。
> 模型:Qwen2.5-1.5B-Instruct + 4bit QLoRA(仅训练 **1.18%** 参数,18.5M / 1.56B)。
> 本文记录评测方法,以及两轮「读错误清单 → 定向补数据 → 再训」的迭代过程。原始日志见同目录 `sft_eval_diag.log`(首轮)与 `sft_eval2.log`(迭代后)。

## 1. 评测协议(三指标 + 防泄漏)

三个指标从严到宽,分别定位不同层级的问题:

| 指标 | 含义 | 定位的能力 |
|------|------|-----------|
| 完全匹配率 | 函数名 + 全部参数完全正确 | 端到端可用性 |
| 函数名正确率 | 只看函数是否选对 | 意图识别 |
| 意图(宽口径) | 容忍参数/格式差异的意图判定 | 语义理解下限 |

**防泄漏与防过拟合:**
- 按**措辞模板**切 train / val:同一指令的不同说法不跨集,杜绝「背答案」式虚高
- val 共 54 条,**全程只评一次额度、不反复刷**,避免对测试集过拟合

## 2. 两轮迭代结果(val 54 条)

| | 完全匹配率 | 函数名 | 意图(宽口径) |
|---|---|---|---|
| Base(未微调) | 0.0% | 0.0% | 9.3% |
| 首轮 SFT(117 条) | 74.1% | 88.9% | 83.3% |
| **迭代后 SFT(187 条)** | **94.4%** | **96.3%** | **96.3%** |

> Base 全 0 是**故意用严格 JSON schema 造出来的**:base 会输出 `setDriverSideWindow`、` ```json ` 代码块、中文函数名 `拨打电话`、甚至瞎编电话号码——在固定 10 函数的严格约束下必然失败,从而把「意图」与「参数合规」拆成可独立优化的两个指标。

## 3. 错误归因 → 修复(首轮 → 迭代后)

逐函数准确率的变化,直接证明「补数据」是**定向的、有效的**,而非盲目堆量:

| 错误类型 | 首轮表现 | 根因 | 修复动作 | 迭代后 |
|---|---|---|---|---|
| 风速指令完全不会 | `climate.set_fan_speed` **0/4** | 训练集缺该函数样本 | 补「风速设成 N」系列 | **4/4** ✅ |
| 音量参数名错 | `media.set_volume` 输出 `percent`,gold 是 `value` | schema 不一致样本太少 | 补统一 `value` 的样本 | **5/5** ✅ |
| 拒答边界 | `system.reject` **2/4** | 越界请求样本少、属开放集 | 补越界样本 | 2/4(仍是难点) |

## 4. 仍存在的局限(诚实记录)

- **拒答泛化难**:「推荐一部电影」「帮我写封请假邮件」仍被误判为 `media.control` 或幻觉出 `mail.reject` 函数——`system.reject` 是开放集,靠补样本收敛慢。下一步:用「兜底规则 + 置信度阈值」而非纯靠 SFT。
- **反向语义未根治**:「所有车窗都升上去」应为 `percent:0`,模型两轮都输出 `percent:100`——没建立「升=关=0」的物理映射,是 1.5B 小模型的能力上限。可在 prompt 中显式给出参数语义说明。

## 5. 定性对比(base vs SFT)

| 指令 | Base 输出 | SFT 输出 |
|---|---|---|
| 我想要 21 度 | `set车内温度 {temperature:21}` | `climate.set_temperature {zone:all, value:21}` ✅ |
| 拨打张经理的电话 | `拨打电话 {phone_number:"13800138000"}`(瞎编号码) | `phone.call {contact:"张经理"}` ✅ |
| 驾驶位弄到 26 度 | `set_driver_seat_temperature {...}` | `climate.set_temperature {zone:driver, value:26}` ✅ |
| 切下一首 | ` ```json {function:"cut_song"} ``` ` | `media.control {action:"next"}` ✅ |
