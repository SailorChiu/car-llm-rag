# car-llm-rag

> 车载场景大模型助手 —— 在消费级硬件（RTX 4060 8G / WSL）上，用 **Qwen2.5-1.5B** 实现「**RAG 问答 + 指令解析 SFT**」两条主线，并以一套可复现的**评测方法论闭环**驱动迭代。

面向「车载大模型 SFT + RAG」方向的求职作品。重点不在于堆模型规模，而在于：**在算力受限的真实约束下，把一个端侧可部署的车载问答系统做扎实，并用数据证明每一步优化的收益。**

---

## ✨ 核心亮点：方法论闭环

很多 LLM 项目「跑通即结束」，本项目的差异点是把工程方法论做实：

| 维度 | 做法 |
|------|------|
| **三指标评测** | 不只看「能不能答」，而是从 **检索召回 / 答案正确性 / 拒答合理性** 三个维度量化，建自有车载 QA 评测集 |
| **防数据泄漏** | 严格切分 train / dev / test，SFT 指令集与评测集去重，避免「背答案」式虚高 |
| **错误驱动迭代** | 每轮按 bad case 归因（检索召回不足 / prompt 不当 / 模型能力不足），定位瓶颈再针对性优化，而非盲目调参 |

## 🏗️ 系统架构

```
用户提问
   │
   ▼
[检索层] 车载知识库 (手册/法规/FAQ) ──► 向量检索 Top-K
   │
   ▼
[Prompt 组装] 检索上下文 + 指令模板
   │
   ▼
[推理层] Qwen2.5-1.5B
   ├─ llama.cpp  (GGUF 量化, 端侧/低显存离线部署)
   └─ vLLM       (服务端高并发, 对照基线)
   │
   ▼
[评测层] 三指标 + bad case 归因  ──►  错误驱动迭代
```

两条独立主线：
- **主线 A — RAG 车载问答**：检索增强，解决「模型不知道这台车 / 这条法规」的问题
- **主线 B — 指令解析 SFT**：QLoRA 微调，把「打开空调到 24 度」类自然语言解析成结构化指令

## 🔧 llama.cpp 量化对照实验（部署核心）

把 FP16 权重转为 GGUF 并做多档量化，从「显存 / 速度 / 质量」三维权衡选型，而非直接拍 Q4。

| 量化档位 | 模型体积 | 显存占用 | 解码速度 (tok/s) | 评测集质量 |
|----------|---------|---------|------------------|-----------|
| FP16     | _待填_  | _待填_  | _待填_           | 基线      |
| Q8_0     | _待填_  | _待填_  | _待填_           | _待填_    |
| Q5_K_M   | _待填_  | _待填_  | _待填_           | _待填_    |
| **Q4_K_M** ✅ | _待填_  | _待填_  | _待填_           | _待填_    |

> 结论：选定 **Q4_K_M** 作为端侧部署档位 —— 以约 _X%_ 的质量损失换取约 _3x_ 体积压缩与显存下降，在 8G 显存下流畅运行。
>
> _（表中 `待填` 为本机实测值占位，benchmark 脚本见 `scripts/`）_
